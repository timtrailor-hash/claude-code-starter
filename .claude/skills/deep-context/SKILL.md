# /deep-context — pre-task context assembly (v3, read-all)

This skill produces a single `context.md` (<50K tokens) that gives the *next* session everything it needs to do a high-stakes task well. No compression layer. No pre-filter. Every past session transcript is read, sharded across parallel Opus agents.

**Invoke only when the task warrants plan mode.** Architectural changes, migrations, multi-file refactors, anything with blast radius. Not for simple edits or one-off questions.

**Cost note.** One invocation reads ~15M tokens across ~20 parallel Opus agents. On a Max subscription this is quota-heavy — concurrent work may throttle during a run. Deliberate trade for full-fidelity coverage.

## Prerequisites

- Prestripped corpus on disk at `~/code/memory_server_data/prestripped/` (generated by `python3 -m deep_context.prestrip_all`).
- `deep_context` Python package installed.

## Arguments

`$ARGUMENTS` is the brief — a specific paragraph describing what the downstream session is going to do.

## Pipeline

### 1. Shard the prestripped corpus

```
RUNID=$(date +%s)-$RANDOM
mkdir -p /tmp/dc_$RUNID
python3 -m deep_context.shard --n 20 --out /tmp/dc_$RUNID/shards
```

This writes 20 shard-manifest files (`shard_00.txt` .. `shard_19.txt`), each listing ~56 prestripped transcript paths totalling ~3MB (~750K tokens). Partitioning is byte-balanced.

### 2. Fan out — 22 agents in parallel

Spawn all 22 agents in a **single message** via the Agent tool. Twenty session agents read transcript shards; one topic agent reads all topics; one code agent greps the codebase.

**Each session agent (i in 0..19):**
- `subagent_type: general-purpose` (needs Write tool to save its output)
- `model: opus` (1M context required)
- Prompt:

```
You are session agent #{i} in a /deep-context fan-out.

BRIEF: {brief}

Your shard manifest is at /tmp/dc_{RUNID}/shards/shard_{ii}.txt — it lists ~56 prestripped session transcripts. Read each listed file. Together they fit within your context window.

From these transcripts, extract every claim that is RELEVANT to the brief. A claim is a single specific sentence. Include:
- Decision rationale (why X was chosen, or why an alternative was rejected)
- Failed attempts with their exact failure mode (error string, command, config)
- Specific identifiers: file paths with line numbers, commit SHAs, error messages, IPs, commands
- Unresolved threads that touch this brief

For each claim, the source field is `session:<8-char-session-id>` — the session_id appears at the top of every prestripped file.

If a transcript has no brief-relevant claims, skip it entirely. Do not pad with generic content.

Write your output to /tmp/dc_{RUNID}/sessions_{ii}.json:
{
  "shard_index": {i},
  "summary": "<one-paragraph synthesis of what this shard revealed for the brief, or empty string if nothing matched>",
  "claims": [{"claim": "...", "source": "session:<8chars>"}, ...],
  "unresolved": [{"claim": "...", "source": "session:<8chars>"}, ...]
}
```

**Topics agent:**
- `subagent_type: Explore`
- Prompt: "Read `~/.claude/projects/-Users-timtrailor-Documents-Claude-code/memory/topics/`, filter to files relevant to this brief: {brief}. Extract load-bearing claims. Write JSON to `/tmp/dc_{RUNID}/topics.json` with shape `{summary, claims:[{claim,source}], unresolved, files_likely}`. Source format: `topic:<filename>`."

**Code agent:**
- `subagent_type: Explore`
- Prompt: "For brief {brief}, use Glob+Grep on `~/code/` to identify files likely touched. Do not read whole files — grep for patterns, read surrounding lines only. Write JSON to `/tmp/dc_{RUNID}/code.json` with shape `{summary, claims:[{claim,source}], files_likely}`. Source format: `code:<path>:<line>`."

Wait for all 22 to finish.

### 3. Assemble fanout.json

```
jq -s '{
  topics: .[0],
  code:   .[1],
  sessions: .[2:]
}' \
  /tmp/dc_$RUNID/topics.json \
  /tmp/dc_$RUNID/code.json \
  /tmp/dc_$RUNID/sessions_*.json \
  > /tmp/dc_$RUNID/fanout.json
```

### 4. Aggregate + synthesise (v3.1)

```
python3 -m deep_context.cli aggregate \
  --brief "$BRIEF" \
  --fanout /tmp/dc_$RUNID/fanout.json \
  --out /tmp/dc_$RUNID/context.md \
  --synthesise
```

`--synthesise` runs a single-agent Opus pass over the deduplicated fan-out that
produces the ACTUAL deliverable: a root-cause synthesis with named
cross-cutting patterns. The raw dedup is still written alongside as
`context.raw.md` for audit.

**Why synthesis matters.** The raw dedup is 50K tokens of claims — excellent
fidelity but scored 1/10 on actionability in an independent Gemini
comparison. A single-session research pass scored 9/10 actionability but
only 7/10 coverage. Fan-out + synthesis scored 10/10 on both. Tim's original
framing called for this: "the master session reviews for inconsistencies
and creates one final 'all you need to know' context file". The fan-out is
the research stage; synthesis is the deliverable.

Leave `--synthesise` off only if you genuinely want the raw claim dump (e.g.
for archival audit — the synthesised file contains all load-bearing claims
anyway).

### 5. Instrumentation

Append one line to `~/.claude/logs/deep-context.jsonl`:

```
{"timestamp": "<ISO>", "run_id": "<id>", "task_brief": "<brief>",
 "shard_count": 20, "agent_count": 22,
 "context_tokens": <from aggregate stats.rough_tokens>,
 "total_session_claims": <sum across shards>}
```

## Handoff

Present `/tmp/dc_$RUNID/context.md`. Ask: "Context assembled. Spawn a new session with this, or review first?"

## What changed in v3

Previous versions pre-compressed every session into a small markdown summary and retrieved the summary at task time. Audit showed this dropped 60% of load-bearing detail (identifiers, error strings, exact commands) at compression time — and no amount of retrieval could recover what compression had destroyed.

v3 drops the compression layer entirely. Raw transcripts are prestripped (tool-output elision, 10x reduction) and stored on disk. Every `/deep-context` invocation reads all 1100+ transcripts across 20 parallel Opus agents. The model sees raw detail at task time, so there is no compression to lose fidelity to.

## Troubleshooting

- **Shard missing.** Ensure `python3 -m deep_context.prestrip_all` has run at least once. Re-run is idempotent (skips up-to-date files).
- **Agent timeout.** A shard is too large. Raise `--n` on the shard command (e.g., `--n 30`) to reduce per-shard size.
- **Quota hit mid-run.** Reduce concurrency: run shards in 2 waves of 10 instead of 20 parallel.
- **Aggregator output oddly thin.** Check that session_agent JSONs parsed — failed agents write `[]` claims; the aggregator silently accepts.

## What NOT to do

- Do not run `/deep-context` in the session that will execute the task. Spawn a *fresh* session with the context.md.
- Do not edit the generated context.md by hand. If wrong, fix the agent prompts and re-run.
- Do not commit context.md files — per-task ephemera.
